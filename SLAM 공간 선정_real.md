    1. SLAM은 정적환경에서 하고, 장애물 회피 테스트는 Nav2단계에서
3. 통로 폭이 적당한 곳
    1. 너무 넓으면 장애물을 피할 필요성 x
    2. 너무 좁으면 경로를 못찾거나 costmap상 정지
4. 코너가 있는 공간

    1. localization을 위해서 SLAM 지도 자체가 좋아야 함 → 고정 구조물이 많으면 좋음
    2. 벽, 문틀, 기둥, 코너
    3. 유리벽, 거울은 피하는 게 좋음
6. 정적 장애물과 일시적 장애물 (맵에 없는) 분리해서 테스트
    1. 테스트 지점을 분리하는게 좋음
    2. 정적 장애물 구간, 일시적 장애물 구간

- 공간 제한 - 실제 공간 사용하되 가벽 설치 고려
    
    터틀봇 3 LiDAR 유효거리가 약 3-4미터
    → 벽과벽 사이 거리가 3m 내로 들어오는 공간
    
    모든 기둥/벽이 똑같고 배치까지 규칙적이면 로봇은 자기 위치 파악 못함 (상대적인 맥락으로 장애물 인식)

# 가제보 환경에서 sdf → 슬램 시도 기록

실제 제작 예정 환경의 크기 및 통로 폭 확인 완료 / 슬램 실패 

1. 1차 시도 - stl→sdf 직접 변환 + 터틀봇 spawn
- 방식
    - CAD로 뽑은 stl 파일을 직접 참조하는 sdf 모델 
    → home/map.stl → model.sdf 에서 직접 참조→ world 에 불러오기
    - 가제보에서 sdf 먼저 열고, 다른 터미널에서 TB3 burger는 spawn 으로 불러오기.

→ 텔레옵 키로 주행 문제 없이 가능했으나, Rviz 상에서 map 생성 실패

1. 2차 시도 - mesh 기반 map + turtlebot3 urdf/mesh 직접 입력
- 방식
    - gazebo models/meshes 구조 이용 → mesh 기반 맵 구성
    - 터틀봇도 urdf 에 공식 mesh 입혀서 불러왔음

→ 가제보 맵 상 시각화 성공했으나 주행 불가 
1과 3처럼 터틀봇이 흔들거리지도 않고, 텔레옵 입력시 바퀴 회전은 하는데 주행 안함
높이/ 바퀴 무게, 마찰 등 설정 건드려봐도 주행 안되고 1과 같이 rviz 생성 실패 

**1차, 2차 모두 slam 시도 : scan, 

1. 3차 시도 - mesh 기반 맵 + turtlebot spawn
    - 방식
    - 1의 spawn 때 터틀봇 주행이 됐었고, 맵은 sdf 직접 변환보다 mesh 기반으로 불러오는 과정이 더 오류가 적었어서 위 방식으로 진행
